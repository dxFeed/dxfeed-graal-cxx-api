29 /**
30  * Manages network connections to @ref DXFeed "feed" or
31  * @ref DXPublisher "publisher". There are per-process (per GraalVM Isolate for now) ready-to-use singleton instances
32  * that are available with DXEndpoint::getInstance() and DXEndpoint::getInstance(Role) methods as well as
33  * factory methods DXEndpoint::create() and DXEndpoint::create(Role), and a number of configuration methods. Advanced
34  * properties can be configured using
35  * @ref DXEndpoint::newBuilder() "newBuilder()".@ref DXEndpoint::Builder::withProperty(const std::string&, const std::string&)
36  * "withProperty(key, value)".@ref DXEndpoint::Builder::build() "build()".
37  *
38  * See DXFeed for details on how to subscribe to symbols and receive events.
39  *
40  * <h3>Endpoint role</h3>
41  *
42  * Each endpoint has a role that is specified on its creation and cannot be changed afterwards.
43  * The default factory method DXEndpoint::create() creates an endpoint with a @ref Role::FEED "FEED" role.
44  * Endpoints with other roles are created with DXEndpoint::create(Role) factory method. Endpoint role is
45  * represented by @ref Role "DXEndpoint::Role" enumeration.
46  *
47  * Endpoint role defines the behavior of its @ref DXEndpoint::connect(const std::string&) "connect" method:
48  *
49  * - @ref Role::FEED "FEED" connects to the remote data feed provider and is optimized for real-time or
50  * delayed data processing (<b>this is a default role</b>).
51  * DXEndpoint::getFeed() method returns a feed object that subscribes to this remote data feed provider and receives events
52  * from it. When event processing threads cannot keep up (don't have enough CPU time), data is dynamically conflated
53  * to minimize latency between received events and their processing time.
54  * For example:
55  * - <b>`DXEndpoint::create()->connect("demo.dxfeed.com:7300")->getFeed()`</b> returns a demo feed from dxFeed with
56  * sample market quotes.
57  * - <b>`DXEndpoint::create()->connect("localhost:7400")->getFeed()`</b> returns a feed that is connected to a
58  * publisher that is running on the same host. See example below.
59  * - <b>`DXEndpoint::create()->connect("file:demo-sample.data")->getFeed()`</b> returns a feed that is connected to
60  * a "demo-sample.data" file and plays back it as if it was received in real time.
61  *
62  * This endpoint is automatically connected to the configured data feed as explained in default properties section.
63  * - @ref Role::ON_DEMAND_FEED "ON_DEMAND_FEED" is similar to @ref Role::FEED "FEED", but it is designed to be used with
64  * OnDemandService for historical data replay only. It is configured with default properties, but is not connected
65  * automatically to the data provider until @ref OnDemandService::replay(Date, double) "OnDemandService->replay" method is invoked.
66  * - @ref Role::STREAM_FEED "STREAM_FEED" is similar to @ref Role::FEED "FEED" and also connects to the remote data
67  * feed provider, but is designed for bulk parsing of data from files. DXEndpoint::getFeed() method returns feed
68  * object that subscribes to the data from the opened files and receives events from them. Events from the files are
69  * not conflated and are processed as fast as possible. Note, that in this role, DXFeed::getLastEvent() method does not
70  * work and time-series subscription is not supported.
71  * For example:
72  * ```cpp
73  * auto endpoint = DXEndpoint::create(DXEndpoint::Role::STREAM_FEED);
74  * auto feed = endpoint->getFeed();
75  * ```
76  * creates a feed that is ready to read data from file as soon as the following code is invoked:
77  * ```cpp
78  * endpoint->connect("file:demo-sample.data[speed=max]");
79  * ```
80  * "[speed=max]" clause forces to the file reader to play back all the data from "demo-sample.data" file as fast as
81  * data subscribers are processing it.
82  * - @ref Role::PUBLISHER "PUBLISHER" connects to the remote publisher hub (also known as multiplexor) or creates a
83  * publisher on the local host. DXEndpoint::getPublisher() method returns a publisher object that publishes events to all
84  * connected feeds.
85  * For example: <b>`DXEndpoint->create(DXEndpoint::Role::PUBLISHER)->connect(":7400")->getPublisher()`</b>
86  * returns a publisher that is waiting for connections on TCP/IP port 7400. The published events will be delivered to
87  * all feeds that are connected to this publisher.
88  * This endpoint is automatically connected to the configured data feed as explained in default properties section.
89  * - @ref Role::LOCAL_HUB "LOCAL_HUB" creates a local hub without ability to establish network connections.
90  * Events that are published via @ref DXEndpoint::getPublisher() "publisher" are delivered to local @ref DXEndpoint::getFeed() "feed"
91  * only.
92  *
93  * <h3>Endpoint state</h3>
94  *
95  * Each endpoint has a state that can be retrieved with DXEndpoint::getState() method.
96  * When endpoint is created with any role and default address is not specified in default properties, then it is not
97  * connected to any remote endpoint.
98  * Its state is @ref State::NOT_CONNECTED "NOT_CONNECTED".
99  *
100  * @ref Role::FEED "Feed" and @ref Role::PUBLISHER "publisher" endpoints can connect to remote endpoints of the opposite
101  * role. Connection is initiated by @ref DXEndpoint::connect(const std::string&) "connect" method.
102  * The endpoint state becomes @ref State::CONNECTING "CONNECTING".
103  *
104  * When the actual connection to the remote endpoint is established, the endpoint state becomes
105  * @ref State::CONNECTED "CONNECTED".
106  *
107  * Network connections can temporarily break and return endpoint back into @ref State::CONNECTING "CONNECTING" state.
108  * File connections can be completed and return endpoint into @ref State::NOT_CONNECTED "NOT_CONNECTED" state.
109  *
110  * Connection to the remote endpoint can be terminated with DXEndpoint::disconnect() method.
111  * The endpoint state becomes @ref State::NOT_CONNECTED "NOT_CONNECTED".
112  *
113  * Endpoint can be closed with DXEndpoint::close() method. The endpoint state becomes @ref State::CLOSED "CLOSED". This is a
114  * final state. All connection are terminated and all internal resources that are held by this endpoint are freed.
115  * No further connections can be initiated.
116  *
117  * <h3>Event times</h3>
118  *
119  * The EventType::getEventTime() on received events is available only when the endpoint is created with
120  * DXEndpoint::DXENDPOINT_EVENT_TIME_PROPERTY property and the data source has embedded event times. This is typically true only
121  * for data events that are read from historical tape files (see above) and from OnDemandService.
122  * Events that are coming from a network connections do not have an embedded event time information and
123  * event time is not available for them anyway.
124  *
125  * <h3><a name="defaultPropertiesSection">Default properties</a></h3>
126  *
127  * Default properties are loaded from "dxfeed.properties" or "dxpublisher.properties" file depending on
128  * the @ref Role "role" of created endpoint. "dxfeed.properties" is used for
129  * @ref Role::FEED "FEED" and @ref Role::ON_DEMAND_FEED "ON_DEMAND_FEED",
130  * "dxpublisher.properties" is used for @ref Role::PUBLISHER "PUBLISHER"}.
131  * @ref Role::STREAM_FEED "STREAM_FEED" and @ref Role::LOCAL_HUB "LOCAL_HUB" do not support properties file.
132  *
133  * The location of this file can be specified using
134  * @ref Builder::withProperty(const std::string&, const std::string&) "withProperty"(::DXFEED_PROPERTIES_PROPERTY, path)
135  * or
136  * @ref Builder::withProperty(const std::string&, const std::string&) "withProperty"(::DXPUBLISHER_PROPERTIES_PROPERTY,
137  * path) correspondingly. When the location of this file is not explicitly specified using
138  * @ref Builder::withProperty(const std::string&, const std::string&) "withProperty" method, then the file path is taken
139  * from a system property with the corresponding name.
140  *
141  * When the path to the above properties file is not provided, then a resource named "dxfeed.properties" or
142  * "dxpublisher.properties" is loaded from classpath. When classpath is set to "." (current directory),
143  * it means that the corresponding file can be placed into the current directory with any need to specify additional
144  * properties.
145  *
146  * Defaults for individual properties can be also provided using system properties when they are not specified
147  * in the configuration file. System properties override configuration loaded from classpath resource, but don't
148  * override configuration from the user-specified configuration file.
149  *
150  * The DXEndpoint::NAME_PROPERTY is the exception from the above rule. It is never loaded from system properties.
151  * It can be only specified in configuration file or programmatically. There is a convenience
152  * @ref Builder::withName(const std::string&) "Builder.withName" method for it. It is recommended to assign short and
153  * meaningful endpoint names when multiple endpoints are used in the same process (one GraalVM Isolate for now).
154  * The name of the endpoint shall describe its role in the particular application.
155  *
156  * Note, that individual properties that are programmatically set using @ref Builder::withProperty(const std::string&,
157  * const std::string&) "withProperty" method always take precedence.
158  *
159  * @ref Role::FEED "FEED" and @ref Role::PUBLISHER "PUBLISHER" automatically establish connection on creation
160  * when the corresponding DXEndpoint::DXFEED_ADDRESS_PROPERTY or DXEndpoint::DXPUBLISHER_ADDRESS_PROPERTY is specified.
161  *
162  * <h3>Permanent subscription</h3>
163  *
164  * Endpoint properties can define permanent subscription for specific sets of symbols and event types in
165  * the data feed, so that DXFeed methods like @ref DXFeed::getLastEventIfSubscribed "getLastEventIfSubscribed",
166  * @ref DXFeed::getIndexedEventsIfSubscribed "getIndexedEventsIfSubscribed", and
167  * @ref DXFeed::getTimeSeriesIfSubscribed "getTimeSeriesIfSubscribed" can be used without a need to create a
168  * separate DXFeedSubscription object. Please, contact dxFeed support for details
169  * on the required configuration.
170  *
171  * <h3>Threads and locks</h3>
172  *
173  * This class is thread-safe and can be used concurrently from multiple threads without external synchronization.
174  *
175  * [Javadoc.](https://docs.dxfeed.com/dxfeed/api/com/dxfeed/api/DXEndpoint.html)
176  */

395  LOCAL_HUB

514  /**
515  * Returns a default application-wide singleton instance of DXEndpoint for a specific role.
516  * Most applications use only a single data-source and should rely on this method to get one.
517  * This method creates an endpoint with the corresponding role on the first use with a default
518  * configuration as explained in
519  * <a href="#defaultPropertiesSection">default properties section</a> of DXEndpoint class documentation.
520  * You can provide configuration via system properties as explained there.
521  *
522  * The configuration does not have to include an address. You can use @ref DXEndpoint::connect(const std::string&)
523  * "connect(address)" and DXEndpoint::disconnect() methods on the instance that is returned by this method to programmatically
524  * establish and tear-down connection to a user-provided address.
525  *
526  * If you need a fully programmatic configuration and/or multiple endpoints of the same role in your
527  * application, then create a custom instance of DXEndpoint with DXEndpoint::newBuilder() method, configure it,
528  * and use Builder::build() method.
529  *
530  * @param role The role of DXEndpoint instance

683  * The endpoint @ref DXEndpoint::getState() "state" immediately becomes @ref State::CONNECTING "CONNECTING" otherwise.

699  * listeners.
