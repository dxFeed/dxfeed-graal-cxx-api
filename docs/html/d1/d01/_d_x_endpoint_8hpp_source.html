29/**
30 * Manages network connections to @ref DXFeed "feed" or
31 * @ref DXPublisher "publisher". There are per-process (per GraalVM Isolate for now) ready-to-use singleton instances
32 * that are available with DXEndpoint::getInstance() and DXEndpoint::getInstance(Role) methods as well as
33 * factory methods DXEndpoint::create() and DXEndpoint::create(Role), and a number of configuration methods. Advanced
34 * properties can be configured using
35 * @ref DXEndpoint::newBuilder() "newBuilder()".@ref DXEndpoint::Builder::withProperty(const std::string&, const
36 * std::string&) "withProperty(key, value)".@ref DXEndpoint::Builder::build() "build()".
37 *
38 * See DXFeed for details on how to subscribe to symbols and receive events.
39 *
40 * <h3>Endpoint role</h3>
41 *
42 * Each endpoint has a role that is specified on its creation and cannot be changed afterwards.
43 * The default factory method DXEndpoint::create() creates an endpoint with a @ref Role::FEED "FEED" role.
44 * Endpoints with other roles are created with DXEndpoint::create(Role) factory method. Endpoint role is
45 * represented by @ref Role "DXEndpoint::Role" enumeration.
46 *
47 * Endpoint role defines the behavior of its @ref DXEndpoint::connect(const std::string&) "connect" method:
48 *
49 * - @ref Role::FEED "FEED" connects to the remote data feed provider and is optimized for real-time or
50 * delayed data processing (<b>this is a default role</b>).
51 * DXEndpoint::getFeed() method returns a feed object that subscribes to this remote data feed provider and receives
52 * events from it. When event processing threads cannot keep up (don't have enough CPU time), data is dynamically
53 * conflated to minimize latency between received events and their processing time. For example:
54 * - <b>`DXEndpoint::create()->connect("demo.dxfeed.com:7300")->getFeed()`</b> returns a demo feed from dxFeed with
55 * sample market quotes.
56 * - <b>`DXEndpoint::create()->connect("localhost:7400")->getFeed()`</b> returns a feed that is connected to a
57 * publisher that is running on the same host. See example below.
58 * - <b>`DXEndpoint::create()->connect("file:demo-sample.data")->getFeed()`</b> returns a feed that is connected to
59 * a "demo-sample.data" file and plays back it as if it was received in real time.
60 *
61 * This endpoint is automatically connected to the configured data feed as explained in default properties section.
62 * - @ref Role::ON_DEMAND_FEED "ON_DEMAND_FEED" is similar to @ref Role::FEED "FEED", but it is designed to be used with
63 * OnDemandService for historical data replay only. It is configured with default properties, but is not connected
64 * automatically to the data provider until @ref OnDemandService::replay(Date, double) "OnDemandService->replay"
65 * method is invoked.
66 * - @ref Role::STREAM_FEED "STREAM_FEED" is similar to @ref Role::FEED "FEED" and also connects to the remote data
67 * feed provider, but is designed for bulk parsing of data from files. DXEndpoint::getFeed() method returns feed
68 * object that subscribes to the data from the opened files and receives events from them. Events from the files are
69 * not conflated and are processed as fast as possible. Note, that in this role, DXFeed::getLastEvent() method does
70 * not work and time-series subscription is not supported. For example:
71 * ```cpp
72 * auto endpoint = DXEndpoint::create(DXEndpoint::Role::STREAM_FEED);
73 * auto feed = endpoint->getFeed();
74 * ```
75 * creates a feed that is ready to read data from file as soon as the following code is invoked:
76 * ```cpp
77 * endpoint->connect("file:demo-sample.data[speed=max]");
78 * ```
79 * "[speed=max]" clause forces to the file reader to play back all the data from "demo-sample.data" file as fast as
80 * data subscribers are processing it.
81 * - @ref Role::PUBLISHER "PUBLISHER" connects to the remote publisher hub (also known as multiplexor) or creates a
82 * publisher on the local host. DXEndpoint::getPublisher() method returns a publisher object that publishes events to
83 * all connected feeds. For example:
84 * <b>`DXEndpoint->create(DXEndpoint::Role::PUBLISHER)->connect(":7400")->getPublisher()`</b> returns a publisher that
85 * is waiting for connections on TCP/IP port 7400. The published events will be delivered to all feeds that are
86 * connected to this publisher. This endpoint is automatically connected to the configured data feed as explained in
87 * default properties section.
88 * - @ref Role::LOCAL_HUB "LOCAL_HUB" creates a local hub without ability to establish network connections.
89 * Events that are published via @ref DXEndpoint::getPublisher() "publisher" are delivered to local @ref
90 * DXEndpoint::getFeed() "feed" only.
91 *
92 * <h3>Endpoint state</h3>
93 *
94 * Each endpoint has a state that can be retrieved with DXEndpoint::getState() method.
95 * When endpoint is created with any role and default address is not specified in default properties, then it is not
96 * connected to any remote endpoint.
97 * Its state is @ref State::NOT_CONNECTED "NOT_CONNECTED".
98 *
99 * @ref Role::FEED "Feed" and @ref Role::PUBLISHER "publisher" endpoints can connect to remote endpoints of the opposite
100 * role. Connection is initiated by @ref DXEndpoint::connect(const std::string&) "connect" method.
101 * The endpoint state becomes @ref State::CONNECTING "CONNECTING".
102 *
103 * When the actual connection to the remote endpoint is established, the endpoint state becomes
104 * @ref State::CONNECTED "CONNECTED".
105 *
106 * Network connections can temporarily break and return endpoint back into @ref State::CONNECTING "CONNECTING" state.
107 * File connections can be completed and return endpoint into @ref State::NOT_CONNECTED "NOT_CONNECTED" state.
108 *
109 * Connection to the remote endpoint can be terminated with DXEndpoint::disconnect() method.
110 * The endpoint state becomes @ref State::NOT_CONNECTED "NOT_CONNECTED".
111 *
112 * Endpoint can be closed with DXEndpoint::close() method. The endpoint state becomes @ref State::CLOSED "CLOSED". This
113 * is a final state. All connection are terminated and all internal resources that are held by this endpoint are freed.
114 * No further connections can be initiated.
115 *
116 * <h3>Event times</h3>
117 *
118 * The EventType::getEventTime() on received events is available only when the endpoint is created with
119 * DXEndpoint::DXENDPOINT_EVENT_TIME_PROPERTY property and the data source has embedded event times. This is typically
120 * true only for data events that are read from historical tape files (see above) and from OnDemandService. Events that
121 * are coming from a network connections do not have an embedded event time information and event time is not available
122 * for them anyway.
123 *
124 * <h3><a name="defaultPropertiesSection">Default properties</a></h3>
125 *
126 * Default properties are loaded from "dxfeed.properties" or "dxpublisher.properties" file depending on
127 * the @ref Role "role" of created endpoint. "dxfeed.properties" is used for
128 * @ref Role::FEED "FEED" and @ref Role::ON_DEMAND_FEED "ON_DEMAND_FEED",
129 * "dxpublisher.properties" is used for @ref Role::PUBLISHER "PUBLISHER"}.
130 * @ref Role::STREAM_FEED "STREAM_FEED" and @ref Role::LOCAL_HUB "LOCAL_HUB" do not support properties file.
131 *
132 * The location of this file can be specified using
133 * @ref Builder::withProperty(const std::string&, const std::string&) "withProperty"(::DXFEED_PROPERTIES_PROPERTY, path)
134 * or
135 * @ref Builder::withProperty(const std::string&, const std::string&) "withProperty"(::DXPUBLISHER_PROPERTIES_PROPERTY,
136 * path) correspondingly. When the location of this file is not explicitly specified using
137 * @ref Builder::withProperty(const std::string&, const std::string&) "withProperty" method, then the file path is taken
138 * from a system property with the corresponding name.
139 *
140 * When the path to the above properties file is not provided, then a resource named "dxfeed.properties" or
141 * "dxpublisher.properties" is loaded from classpath. When classpath is set to "." (current directory),
142 * it means that the corresponding file can be placed into the current directory with any need to specify additional
143 * properties.
144 *
145 * Defaults for individual properties can be also provided using system properties when they are not specified
146 * in the configuration file. System properties override configuration loaded from classpath resource, but don't
147 * override configuration from the user-specified configuration file.
148 *
149 * The DXEndpoint::NAME_PROPERTY is the exception from the above rule. It is never loaded from system properties.
150 * It can be only specified in configuration file or programmatically. There is a convenience
151 * @ref Builder::withName(const std::string&) "Builder.withName" method for it. It is recommended to assign short and
152 * meaningful endpoint names when multiple endpoints are used in the same process (one GraalVM Isolate for now).
153 * The name of the endpoint shall describe its role in the particular application.
154 *
155 * Note, that individual properties that are programmatically set using @ref Builder::withProperty(const std::string&,
156 * const std::string&) "withProperty" method always take precedence.
157 *
158 * @ref Role::FEED "FEED" and @ref Role::PUBLISHER "PUBLISHER" automatically establish connection on creation
159 * when the corresponding DXEndpoint::DXFEED_ADDRESS_PROPERTY or DXEndpoint::DXPUBLISHER_ADDRESS_PROPERTY is specified.
160 *
161 * <h3>Permanent subscription</h3>
162 *
163 * Endpoint properties can define permanent subscription for specific sets of symbols and event types in
164 * the data feed, so that DXFeed methods like @ref DXFeed::getLastEventIfSubscribed "getLastEventIfSubscribed",
165 * @ref DXFeed::getIndexedEventsIfSubscribed "getIndexedEventsIfSubscribed", and
166 * @ref DXFeed::getTimeSeriesIfSubscribed "getTimeSeriesIfSubscribed" can be used without a need to create a
167 * separate DXFeedSubscription object. Please, contact dxFeed support for details
168 * on the required configuration.
169 *
170 * <h3>Threads and locks</h3>
171 *
172 * This class is thread-safe and can be used concurrently from multiple threads without external synchronization.
173 *
174 * [Javadoc.](https://docs.dxfeed.com/dxfeed/api/com/dxfeed/api/DXEndpoint.html)
175 */

550 * Most applications use only a single data-source and should rely on this method to get one.
551 * This method creates an endpoint with the corresponding role on the first use with a default
552 * configuration as explained in
553 * <a href="#defaultPropertiesSection">default properties section</a> of DXEndpoint class documentation.
554 * You can provide configuration via system properties as explained there.
555 *
556 * The configuration does not have to include an address. You can use @ref DXEndpoint::connect(const std::string&)
557 * "connect(address)" and DXEndpoint::disconnect() methods on the instance that is returned by this method to
558 * programmatically establish and tear-down connection to a user-provided address.
559 *
560 * If you need a fully programmatic configuration and/or multiple endpoints of the same role in your
561 * application, then create a custom instance of DXEndpoint with DXEndpoint::newBuilder() method, configure it,

565 * @return The DXEndpoint instance

570 }

573
574 if (INSTANCES.contains(role)) {
575 return INSTANCES[role];
576 }
577
578 return INSTANCES[role] = newBuilder()->withRole(role)->build();
